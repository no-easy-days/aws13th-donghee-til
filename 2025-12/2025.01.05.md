## 클라이언트(Client)
의로인 즉 서비스를 요청하는 소님

## 서버(Server)
서비스를 제공하는 사람 즉 응답하는 직원

## 클라이언트-서버 모델
통신 방식: 요청(Request) -> 처리 -> 응답(Response)
장점: 데이터와 로직이 서버에 집중되어 관리하기 편함(중앙 집중 관리)
단점 서버가 다운되면 모든 서비스 중단(단일 장애)

## DNS(Domain Name System)
IP주소와 도메인을 연결해주는 징검다리 역할

## 작동 방식
1. **사용자**: "[`www.jeff-coding.com`](http://www.jeff-coding.com)에 가고 싶은데, 주소(IP)가 뭐예요?"
   
2. **DNS 해결사 (Resolver)**: "제가 알아봐 드릴게요! 먼저 가장 높은 분(Root)께 여쭤보죠."
   
3. **루트 서버(맨 뒤에 숨은 점)**: "나는 정확한 주소는 모르지만, `.com`을 관리하는 서버는 알고 있어. 거기로 가봐."
   
4. **.com 서버**: "나는 [`jeff-coding.com`](http://jeff-coding.com)을 관리하는 서버 주소를 알고 있어. 거기로 가봐."
   
5. **jeff-coding 관리 서버**: "오, 찾으시는 `www` 주소는 `192.0.2.1`입니다!"
    
6. **DNS 해결사**: "사용자님, 알아왔습니다! 주소는 `192.0.2.1`입니다."
    
7. **사용자**: (해당 IP로 접속 성공!)

## DNS 레코드(주소록의 정보들)
**주소(Address) A**: 가장 기본! 도메인 이름에 해당하는 IP 주소(IPv4)를 알려줍니다. 
 예: [jeff-coding.com](http://jeff-coding.com) = 1.2.3.4 (우리 집 주소는 여기!)

 **별명 (Canonical Name) CNAME**: 도메인의 별명을 지어줍니다. 
 예: [www.jeff.com](http://www.jeff.com)은 [jeff.com](http://jeff.com)이랑 같은 곳이야! (별명 → 본명 연결)

## HTTP/HTTP Protocol

**HTTP:** 클라이언트와 서버가 데이터를 서로 주고 받을때 사용하는 표준 방식(주문서)

**Protocol:** 규약, 약속

## HTTP의 특징
| 특징 | 핵심 키워드 | 비유 | 설명 |
| :--- | :--- | :--- | :--- |
| **무상태성 (Stateless)** | 무상태 (기억 안 함) | 금붕어 / 기억상실증 | "이전 요청을 기억하지 않는다" |
| **비연결성 (Connectionless)** | 비연결 (끊음) | 용건만 간단히 | "한번 요청-응답이 끝나면 연결을 끊는다" |  

HTTP의 가장 큰 특징인 **무상태성** 때문에 우리가 로그인을 유지하기 위해 **쿠키(Cookie)** , **세션(Session)** **토큰(Token)** 이라는 기술을 추가로 사용하게 됩니다.

## 🍪 쿠키 (Cookie)

**정의:** 브라우저(내 컴퓨터)에 저장되는 작은 텍스트 파일입니다.

**역할:** 주로 세션 ID나 토큰을 운반하거나, "오늘 하루 팝업 안 보기" 같은 단순 정보를 저장합니다.

## 🔐 세션 (Session)

**정의:** 사용자의 정보를 **서버의 메모리(창고)** 에 저장하고, 사용자에게는 그 창고를 열 수 있는 **'열쇠(Session ID)'** 만 주는 방식입니다.

**역할:** 보안이 중요한 로그인 정보 등을 서버에서 직접 관리합니다.

## 🎫 토큰 (Token / JWT)

**정의:** 사용자의 정보를 암호화하여 하나의 긴 문자열로 만든 뒤, 사용자에게 넘겨주는 증명서입니다.

**역할:** 서버가 정보를 기억하지 않아도(Stateless), 토큰 자체만 보고 사용자를 인증합니다.

## 세션과 쿠키의 연결 고리 및 핵심
많은 분들이 오해하는 게 **"쿠키랑 세션은 완전히 별개다"** 라고 생각하는 것입니다. 하지만 실제로는 세션 기술을 쓰기 위해서 쿠키를 이용합니다.

**세션 생성:** 로그인하면 서버가 내 정보를 서버 방에 저장합니다.

**ID 발급:** 그 방의 열쇠인 **Session ID (세션 아이디)** 를 만듭니다.

**운반:** 이 열쇠를 쿠키에 담아서 사용자에게 보냅니다.

**저장:** 사용자는 이 **'세션 아이디가 담긴 쿠키'** 를 브라우저에 저장합니다.

**결론:** "세션 방식은 **'세션 ID'라는 열쇠를 '쿠키'라는 봉투에 넣어서** 주고받는 방식이다."


| 비교 항목 | **쿠키 (Cookie)** | **세션 (Session)** | **토큰 (Token)** |
| :--- | :--- | :--- | :--- |
| **비유** | **주머니 속 메모지** | **목욕탕 사물함 열쇠** | **위조 방지 여권** |
| **저장 위치** | **사용자** (브라우저) | **서버** (메모리/DB) | **사용자** (브라우저/앱) |
| **데이터 관리** | 사용자가 가짐 (위험) | **서버**가 가짐 (안전) | 사용자가 가짐 (서명으로 보호) |
| **장점 (Good)** | • 사용하기 쉽다.<br>• 속도가 빠르다.<br>• 만료 기간 설정 가능. | • **보안이 가장 강력함.**<br>• 해킹 시 서버에서 즉시 차단 가능.<br>• 데이터 크기 제한 없음. | • **서버 부담이 없음 (확장성).**<br>• 모바일 앱과 호환성 좋음.<br>• 별도 저장소 필요 없음. |
| **단점 (Bad)** | • **보안에 취약함** (탈취/조작 쉬움).<br>• 저장 용량이 작음 (4KB).<br>• 텍스트만 저장 가능. | • **사용자가 많으면 서버가 터짐.**<br>• 서버를 여러 대 쓸 때 복잡함.<br>• 속도가 비교적 느림. | • 한 번 발급하면 **회수(차단)가 어려움.**<br>• 토큰 길이가 길어질 수 있음.<br>• 데이터가 노출될 수 있음 (암호화 X). |
| **주 사용처** | • "오늘 하루 팝업 닫기"<br>• 비회원 장바구니<br>• 자동 완성 검색어 | • **은행, 관공서**<br>• 결제 시스템<br>(보안 최우선) | • **SNS, 모바일 앱**<br>• 대형 서비스 (넷플릭스 등)<br>(확장성 최우선) |

## HTTP Request/Response

### Request(요청)
클라이언트가 서버에 보내는 메시지입니다.

**Request구조**
```
# ========== Start Line ==========
# POST: 데이터를 전송하겠다는 의미
# /api/login: 로그인을 처리하는 서버 경로
# HTTP/1.1: 사용하는 HTTP 버전
POST /api/login HTTP/1.1

# ========== Headers ==========
# Host: 요청을 받을 서버 주소
Host: api.example.com

# Content-Type: Body에 담긴 데이터가 JSON 형식임을 알림
Content-Type: application/json

# Content-Length: Body의 크기가 52바이트임을 알림
Content-Length: 52

# User-Agent: 요청을 보내는 클라이언트 정보
User-Agent: Mozilla/5.0 (Windows NT 10.0)

# ========== 빈 줄 (Header와 Body 구분) ==========

# ========== Body ==========
# 실제 전송할 데이터 (jeff의 로그인 정보)
{
    "username": "jeff",
    "password": "mypassword123"
}
```

## 주요 HTTP 메서드(Method)

| CRUD | 메서드 | 의미 | 식당 비유 | 설명 |
| :---: | :---: | :---: | :--- | :--- |
| **Create** | GET | 조회 | 📋 메뉴판 보기 | 데이터를 가져옵니다 (읽기 전용) |
| **Read** | POST | 생성 | 📝 새 주문하기 | 새로운 데이터를 서버에 전송합니다 |
| **Update** | PUT | 전체 수정 | 🔄 주문 전체 변경 | 기존 데이터를 완전히 교체합니다 |
| **Update** | PATCH | 부분 수정 | ✏️ 주문 일부 변경 | 데이터의 일부만 수정합니다 |
| **Delete** | DELETE | 삭제 | 🗑️ 주문 취소 | 데이터를 삭제합니다 |

**GET vs POST 핵심 차이**

- **GET**: 데이터를 URL에 담아 전송 (주소창에 보임)
    - 예: `GET /search?keyword=노션`
- **POST**: 데이터를 Body에 담아 전송 (주소창에 안 보임)
    - 로그인, 회원가입 등 민감한 정보 전송에 사용

## Response(응답)
서버가 클라이언트에게 보내는 메시지입니다.

**Response 구조**
```
# ========== Start Line ==========
# HTTP/1.1: HTTP 버전
# 200: 상태 코드 (성공)
# OK: 상태 메시지
HTTP/1.1 200 OK

# ========== Headers ==========
# Content-Type: 응답 데이터가 JSON 형식임을 알림
Content-Type: application/json

# Content-Length: Body의 크기
Content-Length: 89

# Set-Cookie: 브라우저에 세션 쿠키를 저장하라는 지시
Set-Cookie: session=abc123xyz; HttpOnly; Secure

# Date: 응답이 생성된 시간
Date: Sun, 04 Jan 2026 11:30:00 GMT

# ========== 빈 줄 (Header와 Body 구분) ==========

# ========== Body ==========
# 서버가 전달하는 응답 데이터
{
    "status": "success",
    "message": "환영합니다, jeff님!",
    "userId": 12345
}
```

## HTTP Status Code(상태 코드)
상태 코드는 **세 자리 숫자**이며, 첫 번째 숫자가 응답의 종류를 나타냅니다.

**리다이렉션:** 서버가 클라이언트에게 **"네가 찾는 페이지가 여기 없으니, 저쪽 주소로 다시 가봐!"** 라고 안내해 주는 것입니다.

| 분류 | 범위 | 의미 | 비유 |
| :---: | :---: | :--- | :--- |
| **1xx** | 100-199 | 정보 응답 | "알겠어, 처리 중이야" |
| **2xx** | 200-299 | 성공 ✅ | "요청 잘 처리했어!" |
| **3xx** | 300-399 | 리다이렉션 | "다른 곳으로 가봐" |
| **4xx** | 400-499 | 클라이언트 오류 ❌ | "네가 잘못했어" |
| **5xx** | 500-599 | 서버 오류 ❌ | "내가 잘못했어" |

## 자주 쓰는 상태 코드

| 코드 | 이름 | 의미 | 예시 상황 |
| :---: | :--- | :--- | :--- |
| **200** | OK | 요청 성공 | 정상적으로 데이터 조회 |
| **201** | Created | 생성 성공 | 새 회원가입 완료 |
| **204** | No Content | 성공했지만 반환할 데이터 없음 | 삭제 완료 |
| **400** | Bad Request | 잘못된 요청 | 필수 값 누락, 형식 오류 |
| **401** | Unauthorized | 인증 필요 | 로그인 안 함 |
| **403** | Forbidden | 권한 없음 | 접근 권한 부족 |
| **404** | Not Found | 찾을 수 없음 | 존재하지 않는 페이지 |
| **500** | Internal Server Error | 서버 내부 오류 | 서버 버그, 장애 |

## 주요 헤더 

| 헤더 | 의미 | 예시 |
| :--- | :--- | :--- |
| **Host** | 요청 대상 서버 주소 | `Host: www.notion.so` |
| **Content-Type** | 본문의 데이터 형식 | `Content-Type: application/json` |
| **Content-Length** | 본문의 길이 (바이트) | `Content-Length: 348` |
| **Authorization** | 인증 정보 | `Authorization: Bearer abc123` |
| **User-Agent** | 클라이언트 정보 | `User-Agent: Chrome/120.0` |
| **Accept** | 클라이언트가 받고 싶은 형식 | `Accept: text/html` |

**Content-Type 중요한 이유**
서버와 클라이언트가 서로 어떤 형식으로 대화할지 미리 약속하는 것입니다.

같은 데이터도 형식에 따라 해석이 달라집니다:

- `application/json` → JSON으로 파싱
- `text/html` → HTML로 렌더링
- `multipart/form-data` → 파일 업로드용
## HTTP 와 API 차이점

| 구분 | HTTP (The Law) | API (The Rule) |
| :--- | :--- | :--- |
| **한 줄 정의** | 인터넷 공통 통신 규약 | 프로그램 간 기능 명세서 |
| **비유** | 도로교통법, 택배 시스템 | 친구와 정한 암호, 메뉴판 |
| **역할** | 운송 수단 (어떻게 보낼지) | 내용물 (무엇을 할지) |
| **범용성** | 전 세계 공통 | 서비스마다 다름 |
| **명령어 예시** | GET, POST | 지도_보여줘(), 결제_해줘() |

이 둘은 따로 노는 것이 아니라 겹쳐서 작동합니다.

"API라는 [주문서]를 HTTP라는 [봉투]에 담아서 보낸다."

**API:** "나 카카오톡 로그인 시켜줘" (내용 작성)

**HTTP:** 이 내용을 봉투에 담아 **우체국(인터넷)**을 통해 카카오 서버로 전송 (배달)

## REST API, URL

**REST:**
HTTP라는 나라 법을 잘 지키기 위한 가이드라인

**API:**
우리들끼리 정한 규칙

**REST API:**
HTTP라는 나라 법을 좀 더 잘 지키기 위한 우리들끼리의 규칙

| 비교 항목 | 🏛️ HTTP (나라의 법) | 🤙 그냥 API (멋대로 친구) | 🎩 REST API (모범 시민 친구) |
| :--- | :--- | :--- | :--- |
| **비유** | **도로교통법 / 헌법** | **우리끼리 대충 정한 규칙** | **법의 정신을 완벽히 지키는 규칙** |
| **핵심 마인드** | "도구(메서드)를 정의한다."<br>(GET, POST, PUT, DELETE) | "작동만 하면 장땡이지."<br>(편한 대로 막 씀) | "도구의 **원래 용도**대로 쓰자."<br>(법 취지 100% 준수) |
| **조회할 때**<br>(Read) | `GET`을 쓰라고 법에 명시됨. | `POST`로 조회하기도 함.<br>(※ 법 위반은 아니지만 이상함) | 무조건 **`GET`**만 사용함.<br>(조회니까 조회 도구를 씀) |
| **삭제할 때**<br>(Delete) | `DELETE`를 쓰라고 법에 명시됨. | `GET /delete?id=1`<br>(조회 도구로 삭제를 시킴 😱) | 무조건 **`DELETE`**만 사용함.<br>(삭제니까 삭제 도구를 씀) |
| **주소(URL) 스타일** | 규칙 없음. (아무거나 됨) | 동사가 들어감 (지저분함)<br>`/getUser`, `/deleteUser` | **명사**만 깔끔하게 씀<br>`/users` (행동은 메서드로 구분) |
| **처음 본 사람의 반응** | "이게 법이구나." | "얘네 규칙은 설명서 안 보면 모르겠네." | **"설명서 안 봐도 대충 감이 오네!"** |
| **요약** | **기반 시설 (인프라)** | **비효율적인 사용** | **가장 표준적이고 효율적인 사용** |


**URL**
**"HTTP(택배 트럭)가 찾아갈 목적지 주소"** 라고 보시면 됩니다.
```
https://api.example.com:443/users/jeff?role=admin#profile
└──┬─┘ └──────┬───────┘ └┬┘└────┬────┘└────┬────┘└──┬───┘
 스킴		호스트		포트   경로	      쿼리	  프래그먼트
```
| 구성 요소 | 예시 | 설명 |
| :--- | :--- | :--- |
| **스킴 (Scheme)** | `https` | 통신 프로토콜 (http, https, ftp 등) |
| **호스트 (Host)** | `api.example.com` | 서버의 주소 (도메인 또는 IP) |
| **포트 (Port)** | `443` | 서버의 출입문 번호 (생략 시 기본값 사용) |
| **경로 (Path)** | `/users/jeff` | 자원의 위치 |
| **쿼리 (Query)** | `?role=admin` | 추가 조건 (필터링, 정렬 등) |
| **프래그먼트 (Fragment)** | `#profile` | 페이지 내 특정 위치 |


## URL IP주소 비교
| 구분 | IP 주소 (IP Address) | URL (Uniform Resource Locator) |
| :--- | :--- | :--- |
| **생김새** | `223.130.200.107` (숫자 덩어리) | `https://www.naver.com/index.html` |
| **비유** | **GPS 좌표** (위도, 경도) | **전체 안내문** (교통수단 + 건물명 + 방 번호) |
| **누가 좋아함?** | **컴퓨터 / 기계** (숫자라 처리가 빠름) | **사람** (의미를 알기 쉬움) |
| **범위** | 딱 **서버(건물)** 위치만 가리킴 | 서버 위치 + **파일 위치(리소스)**까지 가리킴 |
| **변경 가능성** | 이사 가면 숫자가 바뀜 | 도메인만 그대로라면 주소는 안 바뀜 |

## Json
 JavaScript Object Notation의 약자로, 데이터를 저장하고 전송하기 위한 **텍스트 기반의 데이터 형식**입니다.

| 항목 | 내용 |
| :--- | :--- |
| **정의** | 데이터를 저장하고 전송하기 위한 텍스트 기반 형식 |
| **기본 구조** | `{"키": 값}` 형태의 키-값 쌍 |
| **지원 타입** | 문자열, 숫자, 불리언, null, 배열, 객체 |
| **필수 규칙** | 키는 큰따옴표(`"`) 필수, 마지막 쉼표 금지, 주석 불가 |
| **주요 용도** | 웹 API 통신, 설정 파일, 데이터 저장 |
