## 📝 배운 내용 (Today I Learned)
### 클라이언트(Client)
의로인 즉 서비스를 요청하는 소님
### 서버(Server)
서비스를 제공하는 사람 즉 응답하는 직원
### 클라이언트-서버 모델
통신 방식: 요청(Request) -> 처리 -> 응답(Response)
장점: 데이터와 로직이 서버에 집중되어 관리하기 편함(중앙 집중 관리)
단점 서버가 다운되면 모든 서비스 중단(단일 장애)
### DNS(Domain Name System)
IP주소와 도메인을 연결해주는 징검다리 역할
### 작동 방식
1. **사용자**: "[`www.jeff-coding.com`](http://www.jeff-coding.com)에 가고 싶은데, 주소(IP)가 뭐예요?"
   
2. **DNS 해결사 (Resolver)**: "제가 알아봐 드릴게요! 먼저 가장 높은 분(Root)께 여쭤보죠."
   
3. **루트 서버(맨 뒤에 숨은 점)**: "나는 정확한 주소는 모르지만, `.com`을 관리하는 서버는 알고 있어. 거기로 가봐."
   
4. **.com 서버**: "나는 [`jeff-coding.com`](http://jeff-coding.com)을 관리하는 서버 주소를 알고 있어. 거기로 가봐."
   
5. **jeff-coding 관리 서버**: "오, 찾으시는 `www` 주소는 `192.0.2.1`입니다!"
    
6. **DNS 해결사**: "사용자님, 알아왔습니다! 주소는 `192.0.2.1`입니다."
    
7. **사용자**: (해당 IP로 접속 성공!)
### DNS 레코드(주소록의 정보들)
**주소(Address) A**: 가장 기본! 도메인 이름에 해당하는 IP 주소(IPv4)를 알려줍니다. 
 예: [jeff-coding.com](http://jeff-coding.com) = 1.2.3.4 (우리 집 주소는 여기!)

 **별명 (Canonical Name) CNAME**: 도메인의 별명을 지어줍니다. 
 예: [www.jeff.com](http://www.jeff.com)은 [jeff.com](http://jeff.com)이랑 같은 곳이야! (별명 → 본명 연결)

### IP
인터넷상의 주소

### Port
데이터가 컴퓨터 내의 올바른 프로그램에 도착하기 위한 문(Gate)이다.
IP가 건물 주소라면 port는 건물 내부 주소이다 예) 7강의실 또는 702호 

**포트 번호의 분류**
| 포트 번호 | 서비스(용도) | 설명 |
| :---: | :--- | :--- |
| 80 | HTTP (웹) | 일반적인 웹사이트 접속할 때 쓰는 문 |
| 443 | HTTPS (보안 웹) | 자물쇠 달린 보안 웹사이트 (네이버, 구글 등 대부분) |
| 22 | SSH | 서버 관리자가 원격으로 접속하는 뒷문 |
| 53 | DNS | DNS가 사용하는 문! |
| 3306 | MySQL | 데이터베이스(DB)가 사는 문 |

### HTTP/HTTP Protocol
**HTTP:** 클라이언트와 서버가 데이터를 서로 주고 받을때 사용하는 표준 방식
**Protocol:** 규약, 약속

### HTTP의 특징
| 특징 | 핵심 키워드 | 비유 | 설명 |
| :--- | :--- | :--- | :--- |
| **무상태성 (Stateless)** | 무상태 (기억 안 함) | 금붕어 / 기억상실증 | "이전 요청을 기억하지 않는다" |
| **비연결성 (Connectionless)** | 비연결 (끊음) | 용건만 간단히 | "한번 요청-응답이 끝나면 연결을 끊는다" |  

HTTP의 가장 큰 특징인 **무상태성** 때문에 우리가 로그인을 유지하기 위해 **쿠키(Cookie)** 와 **세션(Session)** 이라는 기술을 추가로 사용하게 됩니다.

### 쿠키(Cookie)
웹 사이트에 접속할 때, 서버가 사용자의 브라우저에 심어놓는 작은 텍스트 파일이고 정보를 사용자(클라이언트)가 가지고 다닌다.

**저장 위치:** **사용자의 컴퓨터(브라우저)** 에 저장됩니다.

**전송:** 사용자가 해당 사이트를 방문할 때마다, 브라우저는 자동으로 이 쿠키를 요청 헤더(Header)에 담아서 서버로 보냅니다.

**주로 쓰는 곳:**

- "아이디 저장" 체크박스

- 쇼핑몰 "오늘 하루 이 창을 보지 않음"

- 비회원 장바구니

- **단점:** 내 컴퓨터에 저장되므로, 남이 훔쳐보거나(보안 취약) 내가 내용을 조작할 수도 있습니다.


### 세션(Session)
사용자가 웹 사이트에 접속해 있는 동안 상태 정보를 서버에 저장하는 기술이다.

**저장 위치:** 서버의 메모리나 데이터베이스에 저장됩니다.

**작동 방식:**

- 서버는 각 사용자에게 **고유한 ID (Session ID)** 를 발급합니다.

- 사용자는 이 ID만 가지고 있고, 실제 데이터는 서버가 가지고 있습니다.

**보안:** 데이터 자체가 서버에 있으므로 쿠키보다 훨씬 안전합니다.

**수명:** 보통 브라우저를 닫으면 세션도 끊깁니다(삭제됩니다).

### 세션과 쿠키의 연결 고리 및 핵심
많은 분들이 오해하는 게 **"쿠키랑 세션은 완전히 별개다"** 라고 생각하는 것입니다. 하지만 실제로는 세션 기술을 쓰기 위해서 쿠키를 이용합니다.

**세션 생성:** 로그인하면 서버가 내 정보를 서버 방에 저장합니다.

**ID 발급:** 그 방의 열쇠인 **Session ID (세션 아이디)** 를 만듭니다.

**운반:** 이 열쇠를 쿠키에 담아서 사용자에게 보냅니다.

**저장:** 사용자는 이 **'세션 아이디가 담긴 쿠키'** 를 브라우저에 저장합니다.

**결론:** "세션 방식은 **'세션 ID'라는 열쇠를 '쿠키'라는 봉투에 넣어서** 주고받는 방식이다."
